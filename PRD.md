# Luna PRD — Ralph Loop用タスク一覧

## 概要
キャバクラ・ナイトクラブ向けPOS SaaS。βリリース: 2026-03-29（新宿アップス）
このPRDはRalph Loopで自律実行するためのタスク一覧。各タスクは機械的に検証可能な完了条件を持つ。

## 実行ルール
- 1タスク完了ごとにコミット
- CLAUDE.mdのルールに従う（日本語、確認不要、pushまで）
- 完了したタスクは `[x]` に変更
- 検証コマンドが通ったら完了とみなす
- 横断要件（セキュリティ・設計原則・パフォーマンス・DB効率）を全タスクで遵守する

#### 1.2.3 SyncEngine 本体
- [ ] `SyncEngine` クラスを作成（シングルトン or 環境オブジェクト）
- [ ] `.networkRestored` 通知を受信したら同期開始
- [ ] アプリフォアグラウンド復帰時にも同期を試行
- [ ] 同期順序: Visit → Nomination → OrderItem → Payment → CashWithdrawal（外部キー依存順）
- [ ] `needsSync == true` のレコードを `created_at` 昇順で取得（古い順に同期）
- [ ] 1レコードずつ同期し、成功したら即座に `needsSync = false` + `lastSyncedAt = now` に更新
- [ ] 同期中の進捗を `@Published syncProgress: (completed: Int, total: Int)` で公開
- [ ] 同期中にアプリがバックグラウンドに移行した場合: `BGTaskScheduler` で継続（最大30秒）
- **検証**: ビルド成功 + SyncEngine のコンパイル通過

#### 1.2.4 リトライ機構
- [ ] 同期失敗時に `syncRetryCount += 1`
- [ ] リトライ間隔: 指数バックオフ（1回目: 5秒後、2回目: 30秒後、3回目: 5分後）
- [ ] `syncRetryCount >= 3` のレコードは同期対象から除外し、`syncFailed` フラグを立てる
- [ ] `syncFailed` レコードが存在する場合、UIに「同期失敗あり」を通知
- [ ] 手動リトライ時は `syncRetryCount` をリセットして再試行
- **検証**: ビルド成功

#### 1.2.5 競合解決
- [ ] サーバー側の `updated_at` とローカルの `updated_at` を比較
- [ ] ローカルが新しい場合: ローカルで上書き（Last Write Wins）
- [ ] サーバーが新しい場合: サーバーのデータを取得してローカルを更新
- [ ] 競合発生時にログ出力（どちらが採用されたかを記録）
- [ ] 会計データ（Payment）は競合時にサーバー優先（金銭データの安全性確保）
- **検証**: ビルド成功

- [ ] 同期順序テスト: 外部キー依存順（Visit→Nomination→...）で同期されること

## 横断要件（全フェーズ共通）

### セキュリティ（OWASP Top 10準拠）
- SQLインジェクション防止: Supabase SDKのパラメータバインディングを必ず使用（生SQL禁止）
- XSS防止: ユーザー入力をHTMLに出力する際は必ずエスケープ（Reactのデフォルト挙動に従う、`dangerouslySetInnerHTML`禁止）
- 認証・認可: Supabase Auth + RLS（tenant_id）で全テーブルをガード。APIルートにも認証チェック必須
- CSRF対策: SameSite Cookie + Supabase Authのトークン管理に準拠
- 機密情報: `.env` / APIキーのコミット禁止。環境変数はVercel/Supabaseの管理画面で設定

### 設計原則
- DDD/クリーンアーキテクチャの原則を全サービスで統一する
- レイヤー分離: UI → Application → Domain → Infrastructure の依存方向を守る
- ビジネスロジック（料金計算・会計処理等）はドメイン層に集約し、UIやDB実装に依存させない

### パフォーマンス
- iPadアプリ: 画面遷移・会計操作のレスポンスは体感0.5秒以内
- Web（HP/管理画面/キャスト画面）: Core Web Vitals達成（LCP < 2.5s, CLS < 0.1, INP < 200ms）
- API応答: Supabaseクエリは200ms以内を目標

### DB効率
- インデックス: `tenant_id`、`created_at`、外部キー、検索対象カラムには必ずインデックスを設定
- N+1回避: リスト表示では関連データをJOINまたは一括取得する（ループ内クエリ禁止）
- SELECT指定: `select(*)` を避け、必要なカラムのみ取得する
- ページネーション: 一覧系は必ずページネーション or 無限スクロールで取得件数を制限する
