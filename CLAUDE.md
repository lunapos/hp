# Luna — プロジェクト横断ルール

## プロジェクト構成

```
luna/
├── lunapos-floor/    ← POS フロア画面（Swift + SwiftUI, iPad）
├── lunapos-cast/     ← キャスト向け画面（Vite + React + TS）
├── lunapos-admin/    ← オーナー管理画面（Vite + React + TS）
├── hp/               ← ホームページ（Next.js）→ lunapos.jp
├── lp/               ← ランディングページ（Vite）→ lp.lunapos.jp
├── supabase/         ← 共有バックエンド（DB / Auth / Realtime）
├── marketing/article/ ← オウンドメディア記事
├── SNS運用/          ← SNS運用ガイド
├── 事業運営/          ← 事業運営タスク管理
├── 規約/             ← 法務文書（弁護士レビュー前ドラフト）
├── images/           ← ロゴ・素材
└── partner/          ← パートナープログラム
```

## 共有バックエンド（Supabase）

全アプリが同一のSupabase DBを参照する。スキーマは `supabase/migrations/` で管理。

## 共通ルール

- **言語**: 日本語で作業・出力（コード内コメントも日本語可）
- **金額**: 全て税込で記載し、表示時は「（税込）」を必ず付記する
- **日付**: YYYY-MM-DD 形式
- **コミットメッセージ**: 日本語、簡潔に
- **秘密情報**: `.env`, credentials, API keyは絶対にコミットしない
- **DB変更**: 必ず `supabase/migrations/` にマイグレーションファイルを作成。直接DB操作しない

### ワークフロー

- **確認不要**: luna 配下のプロジェクトではコマンド実行・push 含め一切確認を求めず自律的に進める
- **「リリースして」= push まで含む**: コミット → push まで一気にやる
- **記事リリース**: article でリリース済みフォルダに移動（ローカル git のみ）→ `hp/content/column/` に MDX 変換して配置 → hp でコミット → push
- **HP 開発ニュース（自動更新必須）**: 全プロジェクト（Floor/Admin/Cast/HP/LP/記事）で機能追加・変更を push するたびに、`hp/src/data/news-updates.ts` に開発アップデートを追加して push する。別途指示がなくても自動で行うこと

### 文体・トーン（マーケティング・記事・SNS）

- 同じ目線で語るトーンを使う
- 上から目線・偉そうな口調はNG（「正直に言います」「はっきり言います」「でも、今はどうですか？」など）
- 断言できないことは断言しない
- 競合の名指し・比較・批判はしない
- Fund / Career を「リリース済み」と誤認させない（「開発ロードマップとして進行中」と表現）

### ブラッシュアップ基準（全アウトプット共通）

**全ての成果物（コード・文章・FAQ・UI テキスト等）を提出する前に、以下の視点で自己レビューを行うこと。**

#### ビジネスモデルとの整合性
- フリーミアム・セルフサーブなのに「お申し込み」「営業日」など有人対応を前提とした表現がないか
- 無料/有料の境界（POS操作 vs 管理画面）が正確に反映されているか
- 未実装・検討中の機能を「ある」と断言していないか（「検討中」「準備を進めています」と書く）

#### 主語と権限の正確さ
- 「オーナー」ではなく「管理画面」など、人ではなくシステムの権限として表現する
- 将来の機能拡張（Career等）で意味が変わる表現を避ける

#### わかりやすさ
- 技術用語を使わず素人にも伝わる言葉で書く
- 抽象的な説明より具体例で伝える（Netflix/Spotify のような身近な例）
- 「〜できます」で終わらず、ユーザーにとっての意味を添える

#### 正直さ・フェアさ
- できないこと・検討中のことは正直に書く
- 無料の対価（データ利活用等）も隠さず明示する
- 過度な期待を持たせない

#### アフィリエイト導線の意識（記事執筆時）
- ASP未登録のため今はリンクを貼らないが、**将来差し替えやすい構造**にしておく
- 具体的なサービス名を出すか「専門業者に依頼がおすすめ」の一文を自然に入れる
- 高相性ジャンル: 店舗物件サイト、会社設立、会計ソフト、害虫駆除、空調クリーニング、水回り修理、ナイトワーク求人
- カラオケ機器のアフィリエイト案件はなし（リース直販が主流）
- 詳細マッピング: `memory/affiliate-strategy.md`

#### セルフレビュー工程（必須）
**コード・文章・あらゆる成果物を提出する前に、「ユーザーがさらに優秀になったバージョン」を模倣して一度レビュー・ブラッシュアップを行ってから提出すること。** 上記の基準に加え、「このまま出したらユーザーに何を突っ込まれるか？」を先回りして直す。

### Ralph Loop開発フロー（PRD駆動の自律開発）

1. **PRD作成**: `PRD.md` にタスク一覧を作成
2. **PRD深掘り**: 各タスクを実装・テスト・検証・エッジケースまで含めたサブタスクに分解
3. **Ralph Loop実行**: PRDのタスクを上から順に消化。全タスクが `[x]` + 検証PASSまで止まらない
4. **完了タスクは即座に `[x]` 更新** → コミット → 次へ

### 進捗レポート出力（必須）

長時間作業中は、ユーザーが非同期で確認できるように以下のフォーマットで出力する:

**タスク完了時:**
```
[完了] Phase X > X.X タスク名
- 実装内容: 概要
- 検証結果: テスト結果 or ビルド結果
- 確認URL: （デプロイがある場合）
- 次のタスク: X.X タスク名
```

**ブロッカー発生時:**
```
[ブロック] Phase X > X.X タスク名
- 原因: 詳細
- 必要なアクション: ユーザーが何をすればいいか
- スキップして次のタスクに進みます
```

**Phase完了時:**
```
[Phase完了] Phase X: フェーズ名
- 完了: X/X タスク
- ブロック: Xタスク（内容）
- 確認URL: （あれば）
- 次のアクション: Phase X に進む or ブロッカー解消？
```

### 記事戦略（SEO優先度）

日本語39記事（+ en/zh各16本の多言語版）でドメイン専門性は十分に積み上がっている。以下を意識して継続:

- **内部リンク網の維持**: 新記事公開時に既存記事からの相互リンクを必ず追加する。特に古い記事→新記事へのリンク追加を忘れない
- **ピラー記事の強化**: 開業ガイド、風営法基礎、市場規模分析など核となる記事は定期的にリライト・最新データ更新する
- **β導入事例**: 新宿アップスのβ運用が安定したら、一次情報ベースの導入事例記事を最優先で書く（E-A-Tの「経験」に直結）
